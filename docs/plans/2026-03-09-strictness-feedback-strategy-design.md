# Strictness × Feedback Wave Generation Strategy Design

## Goal

Redefine Strictness from "DoD analysis depth" to "change tolerance for existing implementations." Enable nextgen wave generation to consider both Strictness level and design-feedback D-Mails when proposing waves, with appropriate response strategies depending on how much existing work is at stake.

## Key Design Decisions

### 1. Strictness Redefinition

**Old:** Controls DoD analysis depth (fog=warning only, alert=sub-issue, lockdown=flag all).

**New:** Controls change tolerance for existing implementations.

```
fog:      No protection for existing implementations. Free to cancel + rebuild.
alert:    Respect existing implementations. Limit blast radius, don't break behavior.
lockdown: Protect existing structure. Prefer additive changes only.
```

DoD checking is ALWAYS full-depth regardless of Strictness. Strictness only affects "how much can we change what's already built."

### 2. Strictness Estimation at Scan Time

Pass 2 (cluster deep scan) has the LLM evaluate issue status distribution + implementation code to produce an estimated Strictness per cluster.

```json
{
  "name": "Auth Module",
  "key": "auth-module",
  "completeness": 0.75,
  "estimated_strictness": "alert",
  "strictness_reasoning": "Done 60%, In Progress 15%. Core auth module has tight coupling.",
  "issues": [...]
}
```

Final Strictness = `max(config value, estimated value)` — always picks the stronger side.

Three-layer resolution:
```
override (manual, per-cluster/label) > estimated (auto, per-cluster) > default (manual)

Final = max(override or default, estimated)
```

### 3. Cluster Key (English Slug)

Clusters get a stable English key alongside the display name:

```go
type ClusterScanResult struct {
    Name                string        `json:"name"`          // display name (language-dependent)
    Key                 string        `json:"key"`           // YAML key (english, slug)
    Completeness        float64       `json:"completeness"`
    EstimatedStrictness string        `json:"estimated_strictness,omitempty"`
    StrictnessReasoning string        `json:"strictness_reasoning,omitempty"`
    Issues              []IssueDetail `json:"issues"`
    // ...existing fields...
}
```

Scan prompt instructs LLM to output both `name` and `key` for each cluster.

### 4. Issue Status in Scan Results

```go
type IssueDetail struct {
    ID           string   `json:"id"`
    Identifier   string   `json:"identifier"`
    Title        string   `json:"title"`
    Status       string   `json:"status"`       // NEW: backlog, todo, in_progress, done, etc.
    Completeness float64  `json:"completeness"`
    Gaps         []string `json:"gaps"`
}
```

Status is output by Claude Code (Linear MCP) during scan. Enables LLM to determine cancel eligibility at nextgen time.

### 5. Estimated Strictness Persistence in Config

Estimated values are written to `sightjack.yaml` after each scan:

```yaml
strictness:
  default: fog
  overrides:
    security: lockdown
    auth-module: alert
  # AUTO-GENERATED by scan. Do not edit manually.
  # Override with 'overrides' section above if you disagree.
  estimated:
    auth-module: alert
    payment-billing: lockdown
```

- Auto-written on scan completion (per-cluster, keyed by cluster key)
- Overwritten on every scan (idempotent)
- To manually override, use the `overrides` section
- Git-tracked: team members see the same estimated values
- Source: Linear issue status + code analysis (external metadata, not local-only)

### 6. Cancel Action

New wave action type: `cancel`

```json
{
  "type": "cancel",
  "issue_id": "MY-123",
  "description": "Approach change required per feedback",
  "detail": "Based on amadeus design-feedback-007: ..."
}
```

Cancel eligibility by Strictness:

| Strictness | Cancellable Status |
|-----------|-------------------|
| fog | Backlog, Todo |
| alert | Backlog, Todo (last resort) |
| lockdown | Backlog only (exceptional) |

In Progress and beyond: cancel NOT allowed at any Strictness level.

Apply execution: Claude Code (Linear MCP) checks status → sets Cancelled → posts reason as comment.

### 7. Nextgen Template Changes

Replace Strictness explanation in `wave_nextgen_{ja,en}.md.tmpl`:

```markdown
## Strictness Level: {{.StrictnessLevel}}

Strictness defines change tolerance for existing implementations:

- fog: No protection. Cancel + rebuild freely per feedback.
- alert: Respect existing. Limit blast radius. Prefer sub-issues and dependency changes.
- lockdown: Protect structure. Additive changes only. Cancel only Backlog issues if unavoidable.

DoD checking: ALWAYS full-depth regardless of Strictness.

## Feedback × Strictness Guidelines

When feedback suggests fundamental changes:
- fog: cancel (Backlog/Todo) + create replacement issues
- alert: preserve existing issues, add sub-issues / update descriptions. Cancel Backlog/Todo as last resort
- lockdown: no cancel except Backlog. Use dependency and DoD additions to steer within existing structure

Note: cancel action requires issue status to be Backlog or Todo. Check the status field in Issue Analysis Results.
```

### 8. Config Type Changes

```go
type StrictnessConfig struct {
    Default   StrictnessLevel            `yaml:"default"`
    Overrides map[string]StrictnessLevel `yaml:"overrides"`
    Estimated map[string]StrictnessLevel `yaml:"estimated"` // auto-generated by scan
}
```

`ResolveStrictness` extended: takes cluster key + labels, resolves across 3 layers, returns max.

### 9. Feedback × Strictness × Rescan Interaction

On rescan after feedback arrival:
- Fresh scan produces new `estimated_strictness` per cluster
- Feedback D-Mails preserved across rescan in FeedbackCollector
- Each cluster's final Strictness may differ, so same feedback produces different wave strategies per cluster

Example:
```
Feedback: "Decouple auth-module from payment-billing"
  Cluster auth-module (alert):  Sub-issues to extract shared interfaces
  Cluster payment-billing (lockdown): Add dependency declarations only
  New cluster api-gateway (fog): Create new issues for decoupled gateway
```

## Impact Summary

| File | Change |
|------|--------|
| `domain/types.go` | IssueDetail.Status, ClusterScanResult.Key/.EstimatedStrictness/.StrictnessReasoning |

| scan prompt templates | Issue status output, estimated strictness output, cluster key output |
| apply prompt templates | Cancel action handling |

| `wave_generator.go` | Cancel action type validation |
| apply handling | Cancel action type execution |
| config I/O | Write estimated section after scan |

## Out of Scope

- Strictness auto-escalation notifications (could alert user when estimated > configured)
- Historical strictness tracking across scans
- Cross-cluster feedback routing (feedback targeting multiple clusters)
