# Sightjack - UX Concept: ゲームのようにIssue群を設計する体験

## このドキュメントの目的

Sightjackが届けるべき**利用体験**を定義する。アーキテクチャ設計書（sightjack-architecture.md）が「何をどう作るか」を記述するのに対し、このドキュメントは「使う人がどう感じ、どう振る舞い、何を得るか」を記述する。

---

## コア体験: SIRENをプレイするようにIssue群を設計する

SIRENのプレイ体験の本質は、**断片的な情報から全体像を組み立てていく知的快感**にある。最初は何が起きているかわからない。Sightjackで敵の視界を覗き、シナリオを1つずつクリアし、行ったり来たりするうちに「あ、あの時のあれが、ここに繋がっていたのか」と全体像が見えてくる。

Sightjackはこの体験をIssue群の設計に再現する。

---

## SIRENプレイ体験のトレース → Sightjack体験への翻訳

### SIRENの体験ループ

```
1. Link Navigatorを開く
   → 時系列×人物のマトリクス。クリア済み/未プレイ/ロック中が一目でわかる。
   → 「全体の何%が見えているか」が感覚的にわかる。

2. 「このキャラのこの時間帯をやろう」と選ぶ
   → 自分の判断で選ぶ。強制されない。
   → 「あのシナリオの続きが気になる」「こっちを先にやった方がいいかも」

3. シナリオに入る
   → Sightjackで周囲を探る。敵の位置、味方の状態、隠されたアイテム。
   → 「ここにこれがあるのか」「この敵、こっちを見てる」

4. 判断して行動する
   → 見えた情報に基づいて自分で決める。「今行くか、待つか」
   → うまくいくこともあれば、「しまった」と思うこともある。

5. 終了条件1をクリア（シナリオクリア）
   → 別のシナリオが解放される。
   → 「おっ、新しいのが開いた」

6. 「あ、終了条件2をやってない」と気づく
   → 前のシナリオに戻る。
   → 「あの場所にあったアーカイブ、取り忘れてた」

7. 別キャラのシナリオで波及に気づく
   → 「あの時、あのキャラがやったことが、ここに影響してたのか！」
   → 断片が繋がる瞬間。最も気持ちいい瞬間。

8. Link Navigatorに戻る
   → 全体像が少し明確になっている。
   → 「次はあっちをやろう」
   → 繰り返し。
```

### Sightjackの体験ループ

```
1. Link Navigatorを開く
   → Cluster × Waveのマトリクス。完了/着手可能/ロック中が一目でわかる。
   → 全体完成度: 45%。「まだ半分以下か」

2. 「Auth ClusterのWave 2をやろう」と選ぶ
   → 自分の判断で選ぶ。AIは解放済みWaveの一覧を見せるだけ。
   → 「認証の設計を先に固めないとAPI側が進められないな」

3. Waveに入る
   → AIがこのClusterのIssue群をSightjackした結果を見せてくる。
   → 「DoD不足が3件、隠れた依存関係が1件、蘇生リスクが1件」
   → 「JWTとSessionの選択が必要です。どうしますか？」

4. 判断して設計する
   → AIの提案を見て、自分で決める。
   → 「JWTで行こう。理由は...」
   → Architect Agentと議論。トレードオフを確認。
   → 「よし、これで決定」

5. Wave完了（Issue群更新 + ADR生成）
   → DoDが追記される。ADR-007が生成される。
   → 別ClusterのWaveが解放される。
   → 「DB Clusterの新しいWaveが開いた。JWTの判断が影響してるな」

6. 「あ、API Clusterの依存関係を修正しないと」と気づく
   → 前のClusterに戻る。
   → 「Auth Clusterで決めたミドルウェア仕様、API側のDoDにも反映しないと」

7. 別ClusterのWaveで波及に気づく
   → DB Clusterのシナリオを進めている時、
   → 「Auth ClusterのJWT判断で、Refresh Tokenのストレージが必要になるのか」
   → 断片が繋がる瞬間。Issue群の全体設計が見えてくる快感。

8. Link Navigatorに戻る
   → 全体完成度: 62%。「だいぶ形になってきた」
   → 「次はFrontend Clusterかな。APIの設計が固まったから」
   → 繰り返し → 75% → 85%
```

## 体験を支える5つの原則

### 1. 「行ったり来たり」を促す、強制しない

SIRENではプレイヤーが自由にシナリオを選べた。「この順番でやれ」と強制されなかった。

Sightjackも同じ。AIは「このWaveが着手可能です」と見せるだけ。人間が「やっぱりあっちに戻ろう」と判断することを歓迎する。

```
? どのWaveに着手しますか？
  1. DB   - Wave 2: トランザクション境界      (50% → 75%)
  2. Front- Wave 1: コンポーネント分割        (28% → 42%)
  3. ← Auth Cluster Wave 1 に戻る            (修正したい箇所がある)
  4. ← 全体のLink Navigatorを見る
```

**戻る選択肢を常に見せる**。前のWaveに戻ることは「間違い」ではなく「設計の深化」である。SIRENでアーカイブを取りに戻るのと同じ。

### 2. 「波及の発見」を演出する

SIRENの最も気持ちいい瞬間は「あの時のあれが、ここに繋がっていた」と気づく瞬間。

Sightjackでは、Wave完了時に**波及の連鎖を明示的に見せる**。

```
[Wave完了] Auth Cluster Wave 2 完了。完成度: 40% → 65%

  🔗 波及:
    → DB Cluster: 新Wave「Refresh Tokenストレージ設計」が生成されました
    → API Cluster: Wave 2のDoDに「ADR-007準拠のトークン検証」が追加されました
    → Frontend Cluster: Wave 1の依存にAuth Wave 2が追加されました

  📊 Link Navigator更新:
    全体完成度: 45% → 52%
    新しく着手可能になったWave: 2件
```

この「波及が見える」瞬間がSightjackの体験の核。自分の設計判断が他のClusterに影響を与え、全体像が動的に変化していく。

### 3. 「全体像が徐々に明確になる」感覚

SIRENのLink Navigatorは、最初はほとんど埋まっていない。プレイするうちにセルが埋まり、全体の物語が見えてくる。

Sightjackでも**完成度の推移を常に表示し、進捗の実感を与える**。

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  SESSION PROGRESS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  開始時:  ████░░░░░░░░░░░░░░░░  30%
  Wave 3:  █████████░░░░░░░░░░░  45%
  Wave 6:  ████████████░░░░░░░░  62%  ← 今ここ
  目標:    █████████████████░░░  85%

  ADR生成: 4件
  Issue更新: 17件
  依存関係追加: 8件
  サブIssue作成: 5件
```

### 4. 「AIが見抜いたもの」への驚き

SIRENでSightjackした時の「うわ、敵がこっち見てる！」という発見の驚き。

Sightjackでは、AIが発見した「人間には見えなかったもの」を明確に演出する。

```
[Sightjack] 👁 API Clusterを分析中...

  ⚡ 発見:
    ENG-201とENG-205が同じDBテーブル (users) を参照しています。
    しかし相互の依存関係が定義されていません。
    同時に下流ツールで実行するとデッドロックのリスクがあります。

  🔄 屍人の蘇生:
    3ヶ月前のENG-089（旧ユーザーAPI）で同様のデッドロックが発生し、
    ホットフィックスを出した履歴があります。
```

この「言われなければ気づかなかった」ものを見せることが、Sightjackの名前の意味そのもの。

### 5. 「設計判断が残る」安心感

SIRENのアーカイブは「なぜこの世界がこうなったのか」を断片的に教えてくれる。

SightjackのADRは「なぜこのIssue群がこう設計されたのか」を記録する。後から実装する人が「なんでこの設計なの？」と疑問に思った時、ADRが答えてくれる。

```
[Scribe] ADR-007 を生成しました

  📝 Title: JWT + Refresh Token方式の採用
  📝 Context: OAuth2連携との統合を考慮
  📝 Decision: JWT（短い有効期限）+ Refresh Token
  📝 Consequences: Refresh Token ストレージが必要

  → ENG-101, ENG-102, ENG-108 にADR参照を追加
  → 将来このIssueに触れる人が「なぜJWTなのか」を即座に理解可能
```

## 体験のアンチパターン（避けるべきもの）

### ❌ 一方通行の進行

```
Wave 1 → Wave 2 → Wave 3 → Wave 4 → 完了
```

これはSIRENの体験ではない。行ったり来たりがない。前に戻れない。パイプライン処理と変わらない。

### ❌ AIが全部決める

```
[AI] 全23 Issueを分析しました。以下のDoD追記とサブIssue作成を提案します。
     承認しますか？ [Y/n]
```

これも体験ではない。人間が設計に参加していない。「AIが作ったIssue群」を承認するだけのスタンプラリー。

### ❌ Issue単位のマイクロ対話

```
ENG-101のDoDを追加しますか？ → はい
ENG-102のDoDを追加しますか？ → はい
ENG-103のDoDを追加しますか？ → はい
...（23回繰り返し）
```

細かすぎる。塊で考えられない。全体像が見えない。

### ❌ 波及が見えない

```
[完了] Auth Cluster Wave 2 完了。
次のWaveを選んでください。
```

波及を見せないと、自分の判断がどう影響したかわからない。SIRENで「別キャラのシナリオが解放された」通知がなかったら、全体の繋がりに気づけない。

### Wave実行中の表示

Wave内では、AIの分析結果が**塊**で提示される。Issue1つずつではなく、Cluster全体の提案をまとめて見せる。

```
━━━ DB Cluster - Wave 2: マイグレーション戦略 + ADR ━━━
  対象: ENG-201, ENG-202, ENG-203  |  現在: 55%  →  目標: 75%

  [Sightjack分析結果]

  📊 このWaveの提案:

  ① 設計判断が必要 (ADR候補: 1件)
     マイグレーション戦略: 段階的デプロイ vs ビッグバン
     → Auth ClusterのADR-007 (JWT) がスキーマ設計に影響
     → Refresh Tokenテーブルの追加が必要

  ② DoD追記 (5件)
     ENG-201: ロールバック手順、既存データ影響範囲
     ENG-202: インデックス戦略の明記
     ENG-203: Refresh Token テーブル定義 (ADR-007由来)

  ③ 依存関係追加 (2件)
     ENG-201 → ENG-202 (スキーマが先、インデックスが後)
     ENG-203は Auth Cluster ENG-102 とのco-dependency

  ④ 屍人の蘇生リスク (1件)
     ENG-089 (旧マイグレーション) でロールバック手順が未定義で
     本番障害が発生した履歴あり

  ────────────────────────────────────────

  ? どうしますか？
    [a] すべて承認  [s] 個別選択  [d] ①について議論  [b] 戻る
```

### 戻る体験

前のWaveに戻って修正する。これは「間違い」ではなく「設計の深化」。

```

> b

  完了済みWave一覧:
    1. Auth  W1: 依存関係整理          (Wave 1で完了)
    2. Auth  W2: 認証方式 DoD + ADR    (Wave 3で完了)
    3. API   W1: エンドポイント分割     (Wave 2で完了)
    4. DB    W1: スキーマ依存整理       (Wave 4で完了)
    5. Front W1: コンポーネント分割     (Wave 5で完了)
    6. Infra W1: 環境構築整理          (Wave 5で完了)

> 3

━━━ API Cluster - Wave 1 (完了済み) を再確認 ━━━

  このWaveで行った変更:
    - ENG-201〜206のエンドポイント分割
    - 依存関係 3件追加

  ⚠ DB Cluster Wave 2の結果を踏まえると、以下の修正が推奨されます:
    - ENG-205 のDoDに「マイグレーション完了後の動作確認」が
      まだ反映されていません（Wave完了後に追加された波及）

  ? 修正を適用しますか？
    [a] 推奨修正を適用  [d] 議論したい  [n] 今は不要  [b] 戻る
```

## 「85%完成」とは何か

全体完成度が85%に達した状態とは：

- 全Clusterの全Issueに**最低限のDoD（終了条件1 + 主要な終了条件2）が定義されている**
- Cluster間の**依存関係が明示されている**
- 主要な**設計判断がADRとして記録されている**
- AI Coding Agentが**各Issueを見て「何をどの順番で実装すればいいか」がわかる**状態

残りの15%は：

- 実装を進める中で初めて見えてくる非機能要件（Fog of War）
- UI/UXの詳細な仕様（プロトタイプを作らないとわからない部分）
- パフォーマンスチューニングの具体的な閾値（負荷テストしないとわからない）
- 他チーム/外部サービスとの統合で発生する調整事項

これらは**設計時に見えないことが正しい**。SIRENでも、全てのアーカイブを1周目で取ることは不可能だった。

## 体験の成功指標

Sightjackが良い体験を提供できているかの指標：

### プレイヤー（利用者）が感じるべきこと

- 「AIが見つけたこの依存関係、自分では気づかなかった」（Sightjackの価値）
- 「この判断がDB Clusterに影響するのか、なるほど」（波及の発見）
- 「前のClusterに戻って修正したら、全体がもっと良くなった」（行ったり来たりの価値）
- 「ADRがあるから、後で実装する時に迷わない」（Archiveの価値）
- 「完成度が上がっていくのが見えて気持ちいい」（進捗の実感）

### プレイヤーが感じてはいけないこと

- 「AIの提案をひたすら承認するだけの作業」（スタンプラリー）
- 「Issue 1つ1つに対して質問に答えるのが面倒」（マイクロ対話）
- 「前に戻れなくて、修正したいのにできない」（一方通行）
- 「AIが何を根拠に提案してるかわからない」（ブラックボックス）
- 「波及が見えなくて、自分の判断の影響がわからない」（断絶感）
