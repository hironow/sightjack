# /sightjack-scan — sightjack designer

Human-initiated entry point. Drives the sightjack MCP server's tools
without ever invoking `claude -p`, so all inference happens inside
this interactive Claude Code session's subscription quota.

`disable-model-invocation: true` makes this skill human-invocable only
— the model cannot auto-trigger it. That is the mechanical form of the
jun15 invariant (LLM work fires only when a human asks).

## Role boundary: design, don't implement

sightjack is the **designer**. One `/sightjack-scan` run scans the
project, designs waves, persists them, and emits specification
d-mails — then stops. **It never applies waves or edits production
code**; implementation belongs to `/expedition-next` (paintress),
which consumes the specification d-mails this skill emits.

## Prerequisites

In a project where `sightjack init` has run, a bare `claude` session
just works: the skill is auto-discovered from `.claude/skills/` and
the MCP server auto-attaches via the project-root `.mcp.json`
(generated by `sightjack mcp-config generate`, merge-aware with the
other tap tools). Manual attach also works:

```bash
claude --mcp-config '{"sightjack":{"command":"sightjack","args":["mcp"]}}'
```

`sightjack mcp` must run from the project root so it can resolve
`.siren/` (config, scan state, event store, outbox).

## Workflow

1. **Verify MCP wiring**. Call `mcp__sightjack__ping`. The tool must
   return `pong`. If it errors, the MCP server is not attached — abort
   and ask the human to run `sightjack mcp-config generate` (or
   relaunch claude with `--mcp-config`).

2. **Scan**. Read the project's issue source and code (Read / Grep /
   Glob; use subagents for breadth — this skill needs no file edits
   and no shell). Group findings into clusters, assess per-cluster
   completeness, and note resurfaced closed-issue patterns (they feed
   `shibito_count`). Pick a stable `session_id` for this scan run
   (e.g. `scan-YYYYMMDD-<short>`); reuse it for every call below.
   If no cluster needs work, skip steps 4-6 and report "no waves
   designed".

3. **Persist the scan result**. Call `mcp__sightjack__save_scan_result`
   with `{session_id, clusters: [...], shibito_count}` (`shibito_count`
   = count of resurfaced closed-issue patterns from step 2; optional).
   Files
   (cluster_*.json read models) land first, then the
   EventScanCompleted ledger entry. If the response says
   `persistence: "files-only"`, re-run the same call once to repair
   (idempotent overwrite); if it still degrades, report and stop.

4. **Design and register waves**. For each cluster that needs work,
   design waves (id, title, description, actions, prerequisites,
   status "available") and call `mcp__sightjack__register_waves` with
   `{session_id, cluster_name, waves}`. Same files-first /
   re-run-to-repair contract as step 3. `next_wave` serves the result
   immediately — verify with one `mcp__sightjack__next_wave` call; if
   it does not serve a registered wave for this session_id, treat it
   as the files-only degradation case (re-run `register_waves` once,
   else report and stop).

5. **Emit specification d-mails**. For each registered wave that is
   ready for implementation, call `mcp__sightjack__dmail` with
   `{kind: "specification", name: "sj-spec-<wave-id>", description,
   body, issues, wave: {id}}`. The tool runs the transactional outbox
   (stage → atomic flush) — phonewave delivers it to the implementer's
   inbox. Re-sending the same name is an idempotent upsert.

6. **Tune strictness when signals appear**. Call
   `mcp__sightjack__update_strictness` with
   `{"level": "fog|alert|lockdown"}` when the scan shows DoD gaps that
   warrant a different enforcement level.

7. **Report**. End with: session_id, clusters saved, waves registered
   (ids), specification d-mails emitted (names), strictness changes,
   and what the human should do next (start the courier if it is not
   running, then `/expedition-next` to implement).

## Failure paths

- **MCP tool error mid-run**: report the tool name and the `reason`
  field, stop. Retry at most once.
- **persistence: "files-only"** from a write tool: re-run that call
  once (idempotent repair). If it degrades again, report and stop —
  the event ledger needs operator attention.
- **dmail rejected** (kind outside produces set / schema violation):
  fix the arguments — never fall back to writing `outbox/` by hand.

## Re-run idempotency

Re-invoking `/sightjack-scan` after a partial run is safe: write tools
overwrite per cluster (same session_id), `dmail` re-sends are upserts
by name, and `update_strictness` is a no-op when unchanged. Start a
repair by calling `mcp__sightjack__get_scan_result` with the previous
`session_id` to see which clusters are already persisted, then re-run
only the missing writes instead of starting a new session.

## What this skill must NOT do

- Invoke `claude -p`, `claude --print`, the Anthropic Agent SDK, or
  any shell wrapper that does so (= billing boundary; semgrep gate
  `.semgrep/jun15-no-headless-llm.yaml` blocks these in production
  code).
- Apply waves or edit production code — that is `/expedition-next`'s
  job (role boundary).
- Write to `outbox/` directly with the Write tool. Direct writes
  bypass the transactional outbox (atomicity / idempotency / OTel
  audit). The `dmail` tool is the canonical emission path.

## Done criteria

A `/sightjack-scan` run is complete when, in a real Claude Code
session with the sightjack MCP server attached:

1. `ping` returns `pong`.
2. `save_scan_result` and `register_waves` return
   `persistence: "files+event-store"`.
3. `next_wave` serves a registered wave for the session_id.
4. Each ready wave has a specification d-mail emitted via `dmail`
   (`persistence: "transactional-outbox"`).
5. The closing report (session / clusters / waves / d-mails / next
   step) is delivered to the human.
